## 3.10 在机器级程序中将控制和数据结合起来

### 3.10.1 理解指针

- 指针类型的强制转换只改变数据的解释方式，不改变地址值本身。比如 `(int *)(p + 7)` 和 `(int *)p + 7` 含义不同：前者先按 `p` 的原类型移动 7 个元素再转换，后者先转换为 `int *` 再移动 7 个 `int` 的距离。
- 指针也可以指向函数，最基本形式为 `返回类型 (*指针名)(参数类型列表)`，若无返回值则返回类型写作 `void`。

### 3.10.3 内存越界引用和缓冲区溢出

- C语言对数组引用不作**边界检查**
- **局部变量**和**状态信息**（如保存的寄存器、返回地址）都存放在栈中

*这两种机制叠加后，可能引发严重的程序错误：对数组**越界写入**时，可能**破坏**栈中存储的关键数据，从而导致程序状态被**覆盖**。*

*当函数随后返回或执行 `ret` 指令时，被破坏的栈状态可能引发**不可预测**的错误甚至程序崩溃。*

*这种典型错误称为**缓冲区溢出（buffer overflow）**。*

- 书上给出一个典型例子：函数中使用**定长字符数组**存储字符串，若输入字符串长度**超过**数组分配空间，就会发生**溢出**并**覆盖**相邻内存区域

```c
void echo(void)
{
    char buf[8];
    gets(buf);
    puts(buf);
}
```

其中 `gets` 向 `buf` 写入数据，此时**栈帧**的结构如下表所示

| 栈帧内容 | 地址方向 |
| :--- | :--- |
| **返回地址 (return addr)** | 高地址 |
| **保存的寄存器 (saved regs)** |  |
| **padding / 对齐填充** |  |
| **局部变量 buf** | 低地址 |

*当输入**超过** `buf` 容量时，写入会从 `buf` 开始持续向高地址方向覆盖，依次破坏上方的对齐填充、`saved regs` 和 `return addr`；如果执行 `ret` 时使用了被篡改的**返回地址**，就会导致：*

- 程序**崩溃**
- 或跳转到**错误**地址
- 或潜在代码执行**劫持**

*缓冲区溢出可导致程序执行非预期代码，甚至被攻击者利用夺取系统**控制权**。攻击者通常通过输入超长字符串，在覆盖栈上数据的同时，将**攻击代码（exploit code）**注入内存，并用该代码的入口地址**覆盖返回地址**，从而在 `ret` 执行时跳转到攻击代码。一种常见做法是直接执行 shell 程序获取系统控制权；另一种则是在执行恶意操作后恢复栈状态，使程序表面上**正常返回**而不被察觉。本质上，攻击利用了栈中**数据与控制信息共存**的结构，将**数据输入**转化为**控制流劫持**手段。*

### 3.10.4 对抗缓冲区溢出攻击

1. **栈随机化**

   - 栈随机化（ASLR 中的一部分）使程序每次运行时**栈地址发生变化**，从而破坏缓冲区溢出攻击中”**预测返回地址并跳转到 shellcode**”的前提，使攻击难度显著提升。
   - 策略是在程序开始时在**栈**上分配 0 到 n 字节之间**随机大小**的空间。

   - 为观察这种变化，可以通过打印**局部变量地址**来发现栈位置在不同运行中的明显波动（32 位下变化较小，64 位下变化更大）。

   ``` c
   int main()
   {
   	long local;
   	printf("local at %p\n",&local);
   	return 0;
   }
   ```

   - 攻击者为了应对这种不确定性，会在 shellcode 前插入大量 NOP（空指令）形成 NOP sled，从而扩大跳转命中的范围，使程序即便无法精确跳转也可能**滑入**（sled）攻击代码并执行

2. **栈破坏检测**

   - C 语言本身无法阻止数组越界，因此 GCC 引入**栈保护机制**（stack protector）

   - 在**局部变量与返回地址之间**插入一个随机**金丝雀值（canary）**，函数入口处从内存读取并保存该值，函数返回前重新检查其是否被修改。
   - canary 值是运行时生成的内存值，将其存放在一个标记为只读的特殊段中，使攻击者无法对其进行覆盖。
   - 如果发生缓冲区溢出并覆盖栈结构，canary 值会被破坏，从而触发 `__stack_chk_fail` 并终止程序。
   - 编译器可通过 `-fno-stack-protector` 关闭该机制，否则会在函数入口和返回阶段自动插入检测代码，从而在返回前发现栈是否被破坏。

3. **限制可执行代码区域**

- 早期 x86 架构中，内存页通常同时具备**读、写、执行**权限，使得**栈上数据也可能被当作代码执行**，从而为缓冲区溢出攻击所利用。现代系统通过虚拟内存机制将内存按页管理，并引入访问权限控制，使代码段通常仅允许执行，数据段仅允许读写。
- 随后，AMD 与 Intel 在 64 位处理器中引入 NX（No-eXecute）位，在硬件层面将”**可写**”和”**可执行**”权限严格分离，阻止数据区代码执行。操作系统配合这一机制实现 W^X（可写与可执行互斥）策略，从而显著降低 shellcode 直接执行的可能性。
- 此外，JIT（即时编译）等技术需要**动态生成代码**，因此系统会在语言或运行时层面上临时放宽部分内存的执行权限，但整体仍依赖受控机制进行管理。
- 这一系列技术（ASLR、canary、NX）共同构成现代缓冲区溢出防护体系，通过**限制可执行内存区域**大幅降低攻击成功率，但仍无法完全消除漏洞利用的风险。

### 3.10.5 支持变长栈帧

*这一节的代码非常实用，包含了压栈、对齐等很规范的写法，我用 CC（Claude Code）添加了注释。*

```c
long vframe(long n, long idx, long *q) {
    long i;
    long *p[n];

    p[0] = &i;
    for (i = 1; i < n; i++)
        p[i] = q;

    return *p[idx];
}
```

```asm
vframe:
    # ─── 帧指针与栈指针的建立 ───
    pushq   %rbp              # 保存旧帧指针
    movq    %rsp, %rbp        # %rbp 固定为当前栈帧基址，此后用 -8(%rbp) 存局部变量 i
    subq    $16, %rsp         # %rsp 下移 16 字节（8 字节存 i，8 字节留作填充）

    # ─── 第一次对齐（16 字节对齐）：为变长数组 p[n] 分配栈空间 ───
    leaq    22(%rdi,8), %rax  # 8*n + 22，多出的 22 包含了预留空间和向上取整的余量
    andq    $-16, %rax        # 把结果向下舍入到 16 的倍数
    subq    %rax, %rsp        # 为数组分配空间，分配后 %rsp 保证 16 字节对齐

    # ─── 第二次对齐（8 字节对齐）：定位数组 p 的起始地址 ───
    leaq    7(%rsp), %rax     # %rsp + 7，+7 是为了向上取整
    shrq    $3, %rax          # 右移 3 位（除以 8）
    leaq    0(,%rax,8), %r8   # 再乘回 8，效果等同于 ceil(%rsp/8)*8，得到 8 字节对齐的地址
    movq    %r8, %rcx         # %rcx = &p[0]

    movq    %rdx, (%rcx)      # p[0] = q
    movq    $1, %rax          # i = 1

.L3:
    movq    %rdx, (%rcx,%rax,8)  # p[i] = q
    addq    $1, %rax
    movq    %rax, -8(%rbp)       # 把 i 存回栈上

.L2:
    movq    -8(%rbp), %rax
    cmpq    %rdi, %rax           # i < n ?
    jl      .L3

    movq    (%rcx,%rsi,8), %rax  # 取 p[idx]（是一个指针）
    movq    (%rax), %rax        # 返回 *p[idx]

    # ─── 恢复栈帧 ───
    leave                   # 等价于 movq %rbp,%rsp + popq %rbp，一步撤销前面的栈帧设置
    ret
```

- 此处 `%rbp` 用作帧指针，后续寻址均基于帧指针而非栈指针进行。
- 栈空间在运行时即时分配。
