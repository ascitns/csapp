## 5.01 优化编译器的能力和局限性

### 5.01.1 GCC 优化级别

| 选项 | 含义 | 特点 |
| :--- | :--- | :--- |
| `-O0` | 不优化 | 编译最快，生成的代码和 C 逐句对应，适合调试 |
| `-Og` | 调试友好优化 | 不干扰调试的优化（不合并变量、不乱序代码），*本书主要用这个* |
| `-O1` | 基本优化 | 在不显著增加编译时间的前提下做优化 |
| `-O2` | 标准优化 | 启用几乎所有不涉及"空间-速度"权衡的优化 |
| `-O3` | 激进优化 | 在 `-O2` 基础上增加内联、循环展开、向量化等 |

- `-O2` 是实际项目中最常用的级别。`-O3` 有时反而会使代码变大变慢，展开和内联增加了指令缓存压力
- 本书大部分示例用 `-Og`，因为反汇编出来的代码和 C 源码对应关系清晰，便于学习

### 5.01.2 编译器能自动完成的优化

| 优化 | 说明 | 实例 |
| :--- | :--- | :--- |
| **寄存器分配** | 把频繁访问的变量放到寄存器 | 循环计数器、累加器自动变寄存器 |
| **代码移动** | 把循环不变量提到循环外面 | 遍历时`strlen(s)` 写在for循环内的时候会重复计算 |
| **强度削弱** | 用廉价指令替代昂贵指令 | `x * 16` → `x << 4`，指令本身的运行速度有差异 |
| **死代码消除** | 删除永不执行的代码 | `if (0) { … }` 直接被删掉 |
| **常量折叠** | 编译期计算常量表达式 | `60 * 60` → `3600`，运行时零开销 |
| **公共子表达式消除** | 重复计算只做一次，结果复用 | `a[i] + a[i]` 只取一次 `a[i]` |

这些优化都是**安全的**，编译器能 100% 确定它们不会改变程序的行为。

- 编译器必须生成**在任何合法输入下都正确**的代码

### 5.01.3 两种阻碍优化的情况

#### 1.内存别名使用（Memory Aliasing）

*两个指针指向**同一块内存**时，它们互为别名。编译器必须假设**任何两个指针都可能互为别名**，除非它能证明不是*

```c

void twiddle1(long *xp, long *yp) {
    *xp += *yp;
    *xp += *yp;
}
```

对应的**汇编**必须读**两**次 `*yp`：

```asm
movq    (%rsi), %rax   
addq    %rax, (%rdi)  
movq    (%rsi), %rax    
addq    %rax, (%rdi)    
```

- 如果编译器自作主张合并为 `*xp += 2 * (*yp)`，就有可能在xp，yp互为别名的情况下得出与原本不同的结果

*如果取一个中间变量读取yp指针所指向的值，可以排除这种情况*

```c
void twiddle2(long *xp, long *yp) {
    long yval = *yp;       // 读到局部变量（寄存器）
    *xp += yval;
    *xp += yval;           // 复用寄存器中的值
}- 
```

- **内存别名显著影响着C 语言性能**，即使是两个完全无关的指针，编译器也必须假设它们可能重叠

#### 2.函数副作用（Side Effects）

- **副作用**是指函数除了返回一个值之外，还做了其他影响程序状态的事情书上列举的是**修改全局变量**的例子

| 副作用类型 | 例子 | 为什么阻碍优化 |
| :--- | :--- | :--- |
| 修改全局变量 | 函数内部 `global_counter++` | 调用次数改变，最终值就改变 |
| 修改传入的指针指向的值 | 函数内部 `*p = xxx` | 可能改到了循环正在读的数据 |
| 写文件 / 打印输出 | `printf`, `fprintf` | 调用次数改变，输出内容就改变 |
| 调用其他有副作用的函数 | 函数内部调了 `rand()` | 同上 |

*书上列举的是**修改全局变量**的例子*

```c
long f(); 
long func1() {
    return f() + f() + f() + f();
}
```

*考虑其修改全局变量的情况*

```c
long counter = 0;           // 全局变量
long f() {
    return ++counter;       // 有副作用：修改全局状态
}
```

- 因为全局变量的值和调用次数相关，所以就不能简单的通过乘法指令优化为4*f（）

- 编译器看不到 `f()` 的实现（在别的 `.c` 文件里）
- C 标准不允许编译器改变**可观测的程序行为**
- 函数如果不在当前编译单元内，编译器就不敢做任何消除调用的优化。
-  **LTO（链接时优化，Link-Time Optimization）**能带来额外性能，因为链接时编译器可以看到**函数全貌**

> 书上给出一种可以通过**内联**（inlining）对包含**函数调用**的代码进行优化的方法，这里的意思与C++**类与对象**那一块学的内联函数相同，旨在**用函数体直接替代函数调用**
>
> 以上面的函数调用为例，可以创建一个局部变量t，通过t+=counter++（*4）的方法**避免了函数调用**
>
> 这样写的好处是**允许编译器进行进一步优化**为更加抽象和直白的行为，例如直接将counter+=4
