## 5.03 程序示例

本章从头到尾用一个例子，对数组所有元素做**合并运算**（求和或求积），逐步优化它。这样 CPE 的变化可以直接归因到每次改动。

### 5.03.1 数据结构与初始版本

```c
// 向量抽象，长度 + 数据指针
typedef struct {
    long len;
    data_t *data;   // data_t 通过 typedef 在四种类型间切换
} vec_rec, *vec_ptr;

// 初始版本，没有做任何优化
void combine1(vec_ptr v, data_t *dest) {
    long i;
    *dest = IDENT;                     // 单位元：加法=0，乘法=1
    for (i = 0; i < vec_length(v); i++) {
        data_t val;
        get_vec_element(v, i, &val);   // 带边界检查的取值函数
        *dest = *dest OP val;          // OP 是宏，可切换为 + 或 *
    }
}
```

- `data_t`、`OP`、`IDENT` 三个宏让同一个代码可以测 8 种组合（4 种类型 × 2 种运算）：

| 宏 | 取值 | 含义 |
| :--- | :--- | :--- |
| `data_t` | `int` / `long` / `float` / `double` | 元素类型 |
| `OP` | `+` 或 `*` | 合并运算 |
| `IDENT` | `0` 或 `1`（对于浮点：`0.0` 或 `1.0`） | 运算的单位元 |

- `get_vec_element` 的实现：

```c
int get_vec_element(vec_ptr v, long index, data_t *dest) {
    if (index < 0 || index >= v->len)   // 边界检查
        return 0;
    *dest = v->data[index];
    return 1;
}
```

以 Intel Core i7 Haswell 为参考，`combine1` 的 CPE 大约在 10 左右，具体值取决于数据类型和运算，但都在 10~11 附近。

- 导致其拖延速度的问题一共有四个

| 问题 | 位置 | C 层面表现 | 汇编层面后果 |
| :--- | :--- | :--- | :--- |
| 循环条件重复调用函数 | `for` 判断表达式 `i < vec_length(v)` | 每轮迭代调一次 `vec_length`，共 n 次 | n 次额外 call + 返回值比较 |
| 循环体内函数调用 | `get_vec_element(v, i, &val)` | 每轮一次 call + 边界检查 | 编译器看不到循环体全貌，无法激进重排 |
| 每轮读/写 `*dest` | `*dest = *dest OP val` | 每轮一次内存读 + 写 | 内存↔寄存器比寄存器↔寄存器慢一个数量级 |
| `*dest` 是指针参数 | — | 编译器不知道 `dest` 是否指向 `data[]` | 别名问题，编译器不敢用寄存器替代 |
